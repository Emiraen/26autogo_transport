## 一、立项依据（项目背景与意义）

### 1.1 桑基鱼塘——千年生态智慧的活态典范

桑基鱼塘起源于珠江三角洲，距今已有 1000 余年历史，是中国先民针对珠三角"低洼易涝"地形进行生态化改造的杰出范例。其核心运作模式可概括为四步循环：

> **塘基种桑 → 桑叶养蚕 → 蚕沙喂鱼 → 塘泥肥桑**

![桑基鱼塘传统循环示意图](https://upload.wikimedia.org/wikipedia/commons/thumb/8/89/Mulberry-dyke_fish-pond_system.svg/640px-Mulberry-dyke_fish-pond_system.svg.png)

*图 1-1 桑基鱼塘"桑—蚕—鱼—泥"四元循环示意（来源：Wikimedia Commons）*

该体系构成了一个近乎封闭的物质循环：

- **桑树**将塘泥中的氮、磷、钾等养分通过根系吸收，以桑叶形式输出生物量；
- **桑叶**作为饲料喂养家蚕，蚕将其转化为蚕茧（经济产物）和蚕沙（蚕粪、残叶）；
- **蚕沙**作为高蛋白饵料投入鱼塘，被草鱼、鲢鱼等摄食，转化为鱼肉与鱼粪；
- **鱼粪、残饵、藻屑**沉积塘底形成富含有机质的塘泥，每年冬季"清淤上基"重施回桑树根部。

整个系统几乎"零废弃"，资源利用率极高，被誉为"中国农业生态智慧的活化石"。

![珠三角桑基鱼塘景观](https://www.fao.org/images/giahs/PHOTOS/China/Pearl-River-Delta_Mulberry-Pond_landscape.jpg)

*图 1-2 珠江三角洲桑基鱼塘景观（来源：FAO GIAHS 官网）*

2017 年 11 月，**"广东佛山基塘农业系统"被联合国粮农组织（FAO）正式列入"全球重要农业文化遗产（GIAHS, Globally Important Agricultural Heritage Systems）"名录**，与浙江青田稻鱼共生系统、云南红河哈尼梯田并列，成为世界农业文化中的重要符号。这是国家、省、市三级农业文化遗产保护体系的重要组成部分。

### 1.2 现实困境

尽管桑基鱼塘具有极高的生态、文化和科学价值，但近 30 年来正面临系统性危机：

| 问题维度 | 具体描述 | 量化数据 |
|---|---|---|
| **城市化挤压** | 工业用地、房地产、道路扩张挤占基塘 | 珠三角面积由 1980s 约 6.7 万亩 → 当前不足 1 万亩，缩减约 85% |
| **传统经验断层** | 投喂量、塘泥施用时机、下基厚度全凭老蚕农经验 | 现存熟练蚕农平均年龄 > 60 岁，年轻人继承率 < 5% |
| **缺乏定量数据** | 桑—蚕—鱼—泥之间的物质流动从未被系统化测量 | 学术界尚无完整的物质流通数据集 |
| **响应迟缓** | 水质恶化通常等"鱼浮头"才能发现 | 一次缺氧事故损失可达每亩 5000–20000 元 |
| **效率低下** | 蚕沙投入与鱼产量最优配比无数据支持 | 估算资源浪费率 15%–30% |

### 1.3 项目意义

| 维度 | 意义阐述 |

## 二、国内外研究现状与发展动态

### 2.1 IoT 在水产养殖领域的应用现状

国内外对池塘水质 IoT 监测的研究已较为成熟：

- **国外**：以色列 AquaMaof、挪威 OxyGuard、日本 Yanmar 推出的智能渔业系统，主要用于工厂化循环水养殖（RAS），监测 DO、pH、温度、氨氮、亚硝酸盐等指标；丹麦 OxyGuard Pacific 系列、美国 In-Situ Aqua TROLL 是国际公认的水质多参数仪标准。
- **国内**：中国农业大学李道亮团队、中国水产科学研究院、上海海洋大学等机构已研发多款池塘水质监测系统。硬件方案以 ESP32/STM32 + LoRa/4G 为主，代表性产品如"渔联网"、"鱼来鱼旺"等；阿里云、腾讯云均推出"数字渔场"行业解决方案。

![IoT水质监测系统典型架构](https://www.researchgate.net/profile/Daoliang-Li/publication/iot-aquaculture-architecture.png)

*图 2-1 典型 IoT 水产养殖监测系统架构（参考李道亮等研究成果）*

**局限性**：仅监测水体本身，未将水质变化与外部输入源（如蚕沙投喂、塘泥扰动）建立定量关联，无法回答"为什么水质恶化"以及"如何优化投喂"。

### 2.2 IoT 在大田/设施农业中的应用现状

土壤墒情、空气温湿度、光照、EC、CO₂ 等传感器在水稻、果园、设施大棚已有大量商业化产品：

- 国内：托普云农、大疆农业、中科原动力、京东农场；
- 国外：John Deere（约翰迪尔）、Trimble Ag、Bosch Plantect。

**局限性**：聚焦于单一作物的种植端，未与下游养殖系统形成耦合，无法处理"种—养"一体化场景。

### 2.3 农业文化遗产（GIAHS）的数字化保护现状

目前国内外针对 GIAHS 的研究多停留在三个层面：

1. **文化层面**：闵庆文团队的《全球重要农业文化遗产》系列著作以记录、描述为主；
2. **生态学层面**：以传统调查法、典型案例分析为主，缺乏长期连续观测；
3. **旅游开发层面**：聚焦"农旅融合"，与生产端脱节。

**少有从"物质循环量化"角度切入的工作**，更鲜有 IoT + 机器学习的综合性研究。本项目正是要填补这一空白。

## 三、项目目标

### 3.1 总体目标

针对珠三角传统桑基鱼塘**缺乏量化监测手段、物质循环效率依赖经验判断**的核心痛点，研发一套覆盖"桑—蚕—鱼—泥"全链路的 IoT 智能监测系统，**首次以传感器数据 + 机器学习方法量化生态循环效率**，并为农户提供可执行的优化建议。

### 3.2 具体目标（可量化）

1. **硬件**：设计并实现 3 类监测节点（鱼塘节点 / 桑基节点 / 蚕房节点），共接入 ≥ 18 路传感器；单节点野外续航 ≥ 30 天；
2. **平台**：构建后端云平台 + Web 管理后台 + 微信小程序前端，支持实时数据查看、历史回溯、阈值告警、远程配置；并发支持 ≥ 100 节点上报；
3. **指标体系**：提出**桑基鱼塘物质循环效率量化指标体系**（≥ 5 项指标），形成行业建议规范初稿；
4. **模型**：训练水质 24h 预测模型，**决定系数 R² ≥ 0.80，DO 预测 MAE ≤ 0.5 mg/L**；
5. **现场验证**：在合作鱼塘**部署运行 ≥ 30 天**，数据完整率 ≥ 95%，告警准确率 ≥ 90%；
6. **成果产出**：发表中文核心或 EI 论文 1 篇 + 软件著作权 1 项 + 实用新型专利 1 项 + 项目结题报告 1 份。

### 4.2 三大监测子系统硬件详细设计

#### 4.2.1 鱼塘监测节点

**主控**：ESP32-WROOM-32E（双核 240 MHz，4 MB Flash，集成 WiFi/BT，多 ADC 通道，支持 Deep-Sleep 微安级电流）。

**传感器清单与配置**：

| 序号 | 传感器型号 | 测量参数 | 量程 / 精度 | 接口 | 供电 | 单价 |
|---|---|---|---|---|---|---|
| S1 | DFRobot SEN0237-A | 溶解氧 DO | 0–20 mg/L / ±0.3 | 模拟电压 → ADC | 5V | ¥350 |
| S2 | DFRobot SEN0161-V2 | pH | 0–14 / ±0.1 | 模拟电压 → ADC | 5V | ¥120 |
| S3 | DS18B20（防水） | 水温 | -55–125 ℃ / ±0.5 | 单总线 (1-Wire) | 3.3V | ¥10 |
| S4 | DFRobot SEN0451 | 电导率 EC | 0–20 mS/cm / ±5% | 模拟电压 → ADC | 5V | ¥250 |
| S5 | DFRobot SEN0189 | 浊度 | 0–4550 NTU | 模拟电压 → ADC | 5V | ¥100 |
| S6 | 工业级氨氮电极 | NH₃-N | 0–100 mg/L | RS-485 (Modbus RTU) | 12V | ¥600 |
| S7 | 超声波液位 JSN-SR04T | 水位 | 25–450 cm / ±1cm | GPIO 触发/回波 | 5V | ¥30 |
| V1 | ESP32-CAM (OV2640) | 视觉/鱼群图像 | 1600×1200 | SPI / WiFi | 5V | ¥45 |

**电源系统**：
- 太阳能板 6V/6W 单晶硅 → CN3791 MPPT 充电管理 → 18650 锂电 3 节并联（约 9000 mAh）→ MP1584 DC-DC 输出 5V/3.3V；
- ESP32 平时 Deep-Sleep（≈ 10 µA），每 5 分钟唤醒采集一次，平均功耗 < 5 mA，电池可支撑无光照 ≥ 7 天；

**结构与防护**：
- IP67 ABS 防水箱（150×100×70 mm）；
- 传感器探头通过 PG7 防水接头引出；
- 浮子式漂浮支架，配尼龙绳系泊；
- TVS 二极管 + 气体放电管做雷击浪涌防护；电源回路加 PTC 自恢复保险丝。

**ESP32 GPIO 分配**：

```
GPIO34 (ADC1_CH6) → DO  (S1, 模拟)
GPIO35 (ADC1_CH7) → pH  (S2, 模拟)
GPIO32 (ADC1_CH4) → EC  (S4, 模拟)
GPIO33 (ADC1_CH5) → 浊度 (S5, 模拟)
GPIO4              → DS18B20 (单总线，需 4.7kΩ 上拉)
GPIO16/17          → UART2 → MAX485 → 氨氮电极 (Modbus RTU)
GPIO12 (Trig) / GPIO13 (Echo) → JSN-SR04T 超声波
GPIO5              → MOSFET 开关，控制传感器电源轨（节能）
GPIO2              → 板载 LED，状态指示
```

**传感器清单**：

| 序号 | 传感器型号 | 参数 | 接口 | 备注 |
|---|---|---|---|---|
| T1 | RS485-SoilMoisture（七合一） | 土壤温度/湿度/EC/pH/N/P/K | RS-485 Modbus RTU | 农业行业标准 |
| T2 | BH1750FVI | 光照强度 | I²C | 1–65535 lx |
| T3 | DHT22 / SHT31 | 空气温湿度 | 单总线 / I²C | 监测桑园小气候 |

**部署**：地下 15 cm 埋深探针 + 地上小型百叶箱；太阳能 5V/3W + 18650 单节锂电（约 3000 mAh，2 周续航）。

| C1 | SHT31-D | 温湿度 | I²C | 蚕房适宜：24–28 ℃，70–85 %RH |
| C2 | SCD41 | CO₂ | I²C | 通风评估 |

| C4 | BH1750FVI | 光照 | I²C | 蚕生长光周期 |

#### 4.2.4 网关（可选）

若部署点 4G 信号弱，则使用一个汇聚网关：

- **主控**：树莓派 4B（4GB） 或 Mixtile Edge2 Kit；
- **接入**：SX1278 LoRa 模块（433 MHz） + 4G Cat.4 模块（中移 ML302）；
- **软件**：运行 Mosquitto MQTT Broker + Node-RED + 本地 SQLite 缓存；
- **作用**：聚合所有节点 → MQTT 上行云端，断网时本地缓存 ≥ 7 天。

### 4.3 通信协议设计

#### 4.3.1 数据帧格式（节点 → 平台）

采用 JSON over MQTT，主题：`gisp/{site_id}/{node_id}/data`。

```json
{
  "node": "fishpond-01",
  "ts": 1730000000,
  "fw": "v1.0.3",
  "bat": 3.92,
  "data": {
    "do": 6.42,
    "ph": 7.15,
    "tw": 26.3,
    "ec": 0.81,
    "ntu": 35.2,
    "nh3": 0.18,
    "level": 132.4
  },
  "meta": { "rssi": -87, "snr": 9.5 }
}
```

#### 4.3.2 控制下行

主题：`gisp/{site_id}/{node_id}/cmd`，支持远程配置采样间隔、阈值、OTA 升级。

#### 4.3.3 LoRa 物理层参数

- 频段：CN470，子带 8 (471.3 MHz 起)；
- SF（扩频因子）：SF9，BW=125 kHz，CR=4/5；
- 单包载荷 ≤ 51 字节（启用 ADR 时空中时长 ≤ 200 ms）；
- 传输功率：14 dBm（合规上限 17 dBm）；
- 链路预算 ≈ 152 dB，可视距 ≥ 5 km，鱼塘环境实测 ≥ 800 m。

### 4.4 边缘端固件架构（PlatformIO + Arduino-ESP32）

```c
    cache_to_flash(d);                    // 边缘缓存（断网续传）
    if (network_ok()) {
        send_mqtt(d);
        flush_cache();
    }
    deep_sleep(SAMPLE_INTERVAL);          // 深度睡眠节能
}
```

**关键技术点**：
- 使用 `esp_sleep_enable_timer_wakeup()` 进入 Deep-Sleep；
- 使用 NVS（Preferences 库）保存配置；使用 LittleFS 做缓存文件；
- 加入 Watchdog（任务看门狗 + 中断看门狗），异常自动重启；
- OTA 通过 ArduinoOTA + AES-128 校验。

### 4.5 物质循环效率量化指标体系（核心创新）

| 编号 | 指标 | 计算公式 | 物理含义 |
|---|---|---|---|
| K1 | 蚕沙-鱼转化率 FCR_sw | 鱼增重(kg) / 蚕沙投入量(kg) | 蚕沙作为饵料的效率，目标 > 0.20 |
| K2 | 塘泥养分指数 MNI | 0.4·Ñ + 0.35·P̃ + 0.25·K̃ | 塘泥肥力综合评分（0–1） |
| K3 | 桑叶产量响应 R_my | Δ桑叶产量 / Δ塘泥施用量 | 塘泥对桑树增产作用，单位 kg/kg |
| K4 | 水质-投喂关联度 ρ_wf | corr(投喂量, ΔDO) | 是否过量投喂，<-0.5 即预警 |
| K5 | 循环闭合度 η_cycle | 实际循环量 / 理论最优循环量 | 系统整体闭合度（首创） |

**η_cycle 详细推导**：

设系统四节点（桑 M, 蚕 W, 鱼 F, 泥 P）之间的物质流向矩阵为 **A**(4×4)，其中 A_ij 为从 i 到 j 的氮当量物质流（kg/月）。理论最优循环（无外源/无废弃）下满足 **质量守恒**：

$$\sum_j A_{ij} = \sum_k A_{ki}, \quad \forall i \in \{M, W, F, P\}$$

定义 **闭合损失** L = ∑|流入 − 流出|；理论最大流量 T_max 由桑叶产量上限给出，则：

$$\eta_{cycle} = 1 - \frac{L}{2 T_{max}} \in [0, 1]$$

η = 1 表示完全闭合；η < 0.6 表示循环严重失衡，建议干预。

### 4.6 主要分析算法

#### 4.6.1 数据预处理

1. **时间对齐**：所有节点对齐到统一 UTC+8 时间轴，5 min 重采样；
2. **缺失值**：< 1 h 缺口用线性插值；> 1 h 用 KNN-imputer（k=5）；
3. **异常值**：3σ 检验 + 物理边界裁剪（如 DO ∈ [0,20]）；
4. **平滑**：DO/pH 使用 **一维卡尔曼滤波**：

```
x_k = x_{k-1} + w,  w ~ N(0, Q)
z_k = x_k + v,      v ~ N(0, R)
K_k = P_k / (P_k + R)
x_k = x_k + K_k * (z_k - x_k)
```

经验取 Q=1e-4，R=1e-2，能在保留趋势的同时滤除高频噪声。

#### 4.6.2 模型 M1：水质 24h 预测（核心算法）

**问题定义**：给定过去 72h（8640 个 5-min 样本）的多变量时序，预测未来 24h 内每小时的 DO、pH、温度。

**对比方案**：

| 方案 | 模型 | 优点 | 缺点 |
|---|---|---|---|
| A | XGBoost (滑窗特征工程) | 训练快、可解释 | 难捕捉长程依赖 |
| B | 单层 LSTM (hidden=64) | 时序基线 | 需要较多数据 |
| C | Seq2Seq LSTM + Attention | 多步预测稳定 | 训练较慢 |
| D | Transformer (Informer) | SOTA | 数据需求大 |

**最终选择**：方案 C 作为主模型，方案 A 作为备选/可解释模型。

**网络结构（方案 C）**：

```
Input (B, 72*12, F=10)
  → Encoder LSTM (hidden=128, 2 layers)
  → Bahdanau Attention
  → Decoder LSTM (hidden=128) → Linear(F=3) → Output (B, 24, 3)
```

**特征工程**（输入维度 F=10）：

- 水质：DO, pH, 水温, EC, 浊度
- 外部：投喂量（蚕沙）、气温、降雨量、气压、太阳辐射

**训练配置**：
- 损失：MSE + 物理一致性正则项 0.1·|Δ DO/Δt - f_dyn|
- 优化器：AdamW，lr=1e-3，cosine schedule
- Batch size=64，Epoch=200，early-stop patience=20
- 训练 / 验证 / 测试 = 70 / 15 / 15
- 数据增强：Jittering（高斯噪声）+ 时序 Mixup

**评估指标**：R²、MAE、RMSE、MAPE，目标 R² ≥ 0.80 (24h DO)。

#### 4.6.3 模型 M2：异常检测（浮头预警）

**算法**：Isolation Forest + LSTM-AutoEncoder 双模型投票。

- Isolation Forest（n_estimators=100）：对每 5min 样本计算异常分数；
- LSTM-AE：编码-解码重建误差 > 阈值（μ+3σ）即视为异常；
- 两者均判异常时触发"红色告警"，仅一者判异常时"黄色提示"。

#### 4.6.4 模型 M3：优化建议生成（决策推理）

输入：当前 K1–K5 指标 + 短期预测；输出：自然语言可执行建议。

**实现方式**：基于规则 + 决策树的混合系统：

```
IF η_cycle < 0.6 AND ρ_wf < -0.5:
    → "投喂量已过量，建议减少 20%"
IF MNI < 0.4 AND 距上次清淤 > 11 个月:
    → "塘泥养分偏低且周期较长，建议安排清淤上基"
IF 预测 6h 内 DO < 3 mg/L:
    → "高风险！建议立即启动增氧机"
```

进阶方案：用 SHAP 值解释 LSTM 预测的关键驱动因子，辅以 LLM (本地 ChatGLM3-6B-int4) 生成自然语言。

#### 4.6.5 模型 M4：桑叶产量回归

**输入特征**：累计积温、累计光照、土壤湿度均值、上次塘泥施用量、施用距今天数；
**模型**：随机森林回归 (n_estimators=200, max_depth=8)；
**目标**：预测下一茬桑叶产量 (kg/亩)，期望 R² ≥ 0.70。

#### 4.6.6 视觉算法：鱼群浮头检测

ESP32-CAM 每 30 min 拍一张水面图像 → 上行至云端 → 服务器使用 **YOLOv8n** 微调模型检测水面"鱼嘴"目标。

- 训练数据：自采 + 网络爬取约 500 张 + Mosaic 增广扩到 5000 张；
- 部署：ONNX Runtime + CPU；
- 预警：单帧检出 > 3 个鱼嘴 → 立即推送告警。
